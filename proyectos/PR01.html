Reconocimiento de red desde Kali Linux

Video

Video de la documentacion del proyecto.

Contexto

En esta práctica se realizó un análisis básico de una máquina dentro de una red local desde Kali Linux. El objetivo fue identificar y encontrar la vulnerabilidades de una maquina virtuale en la que teniamos que lograr entrar y obtener acceso con beneficios de administrador y asi explotar sus vulnerabilidades.

El primer paso fue identificar los dispositivos activos dentro de la red local. Para esto se realizó un escaneo ARP que permite ver qué direcciones IP están activas. 


Currently scanning: Finished!   |   Screen View: Unique Hosts               
                                                                             
 3 Captured ARP Req/Rep packets, from 3 hosts.   Total size: 180             
 _____________________________________________________________________________
   IP            At MAC Address     Count     Len  MAC Vendor / Hostname      
 -----------------------------------------------------------------------------
 192.168.56.1    0a:00:27:00:00:27      1      60  Unknown vendor            
 192.168.56.100  08:00:27:89:5c:85      1      60  PCS Systemtechnik GmbH    
 192.168.56.104  08:00:27:91:e7:0a      1      60  PCS Systemtechnik GmbH

Aquí se identificaron tres hosts dentro de la red. Dos de ellos pertenecen a máquinas virtuales de VirtualBox. Esto fue facil de detectar ya que estabamos trabajando con dos maquinas virtuales de VB.

  • 192.168.56.1 -> Interfaz host (computadora normal)
  • 192.168.56.100 -> Host activo (MV atacante)
  • 192.168.56.104 -> Host a analizar (MV victima)

Escaneo completo de puertos con Nmap

Después del reconocimiento inicial, se realizó un escaneo completo con nmap para identificar servicios abiertos en los hosts encontrados. 


└─$ sudo nmap -p- -sS -sV -sC -A -T4 192.168.56.100

Este comando realiza varias cosas:

  • -p- -> escanea los 65535 puertos
  • -sS -> escaneo SYN (stealth scan)
  • -sV -> detección de versiones de servicios
  • -sC -> ejecuta scripts básicos de nmap
  • -A -> detección agresiva (OS, scripts, traceroute)
  • -T4 -> escaneo más rápido

Aqui me equivoque de direccion IP porque en el comando puse la IP de la maquina atacante (.100) en lugar de la maquina victima (.104). Por lo tanto la salida fue la siguiente: 


All 65535 scanned ports are closed

Esto significa que el host está activo pero no tiene servicios accesibles desde la red.

Escaneo del segundo host

Se realizó el mismo escaneo al host 192.168.56.104 (maquina victima). 


└─$ sudo nmap -p- -sS -sV -sC -A -T4 192.168.56.104

Este host sí tenía varios servicios abiertos:

Puerto Servicio Versión
22 SSH OpenSSH 7.9
80 HTTP nginx 1.14.2
8080 HTTP nginx 1.14.2

Esto indica que el servidor tiene:

  • Acceso remoto por SSH
  • Un servidor web en el puerto 80
  • Otra aplicación web en el puerto 8080

Intento de acceso por SSH

Se intentó conectar directamente al servidor usando SSH con el usuario root. 


└─$ ssh root@192.168.56.104

Sin embargo el acceso fue rechazado porque no se tenía la contraseña correcta. 


Permission denied

Esto indica que el servicio SSH está activo pero requiere autenticación valida.

Pruebas manuales con Netcat

Para entender cómo responde el servidor web también se hicieron pruebas con netcat, que permite enviar peticiones HTTP manualmente. 


└─$ nc 192.168.56.104 80
GET / HTTP/1.1
Host: 192.168.56.104

Esto permite interactuar directamente con el servidor HTTP sin usar navegador.

Consulta del sitio web con Curl

Después se uso curl para ver la respuesta del servidor web. 


└─$ curl http://192.168.56.104

El servidor respondió con una página simple:

Welcome to SNAKEOIL!

Esto confirma que nginx está funcionando correctamente.

Analisis del servicio web en el puerto 8080

Tambien se consultó el servicio en el puerto 8080. 


└─$ curl http://192.168.56.104:8080

Aqui apareció una aplicación web más completa llamada:

Good Tech Inc.'s Snake Oil Project

La página muestra varias secciones como:

  • Introduction
  • House Rules
  • Useful Links
  • Create New Post

Esto sugiere que la aplicación es algún tipo de blog o sistema de posts.

Enumeración de directorios con Gobuster

Para encontrar rutas ocultas dentro del servidor se utilizó la herramienta gobuster. 


└─$ gobuster dir -u http://192.168.56.104:8080 -w /usr/share/wordlists/dirb/common.txt

Resultados encontrados:

  • /create
  • /login
  • /registration
  • /users
  • /run
  • /secret
  • /test

Algunas rutas respondían con códigos interesantes como:

  • 200 -> página existente
  • 405 -> método no permitido
  • 500 -> error interno del servidor

El endpoint /secret devolvía error 500, lo cual podría indicar alguna funcionalidad interna mal configurada por lo tanto era nuestra vulnerabilidad.

Pruebas de la API

Algunas rutas parecian funcionar como API, por ejemplo /login. 


└─$ curl -X POST http://192.168.56.104:8080/login

El servidor respondió indicando que faltaban campos. 


{"message": {"username": "Username field cannot be blank."}}

Se intentó autenticarse con credenciales de prueba. 


└─$ curl -X POST -d "username=patrick&password=test" http://192.168.56.104:8080/login

Respuesta: 


{"message": "Wrong credentials"}

Registro de usuario

La API permitía crear nuevos usuarios usando el endpoint /registration. 


└─$ curl -i -X POST http://192.168.56.104:8080/registration 
-H "Content-Type: application/json" -d '{"username":"rene","password":"123"}'

El servidor respondió creando el usuario y devolviendo un token JWT. 


{"message": "User rene was created. Please use the login API to log in!", "access_token": "eyJ0eX..."}

Este token se puede usar para autenticarse en otras partes de la API.

Uso del token de autenticación

Despues se intento usar el token para acceder al endpoint /secret. 


└─$ curl -H "Authorization: Bearer TOKEN" \
http://192.168.56.104:8080/secret

Sin embargo el servidor seguía respondiendo con: 


{"message": "Internal Server Error"}

Esto puede indicar que:

  • La funcionalidad no está terminada
  • Existe una vulnerabilidad en ese endpoint
  • Requiere otro tipo de autenticación

Conclusión

Durante esta fase del análisis se logró:

  • Identificar hosts activos en la red
  • Detectar servicios abiertos con Nmap
  • Explorar aplicaciones web con curl
  • Enumerar rutas ocultas con gobuster
  • Interactuar con una API web
  • Crear un usuario y obtener un token JWT

El endpoint /secret y el endpoint /run parecen ser los más interesantes para continuar el análisis, ya que responden con errores o comportamientos inesperados que podrían indicar posibles vulnerabilidades.
